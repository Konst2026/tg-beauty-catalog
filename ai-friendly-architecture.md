# AI-Friendly Codebase Architecture
### Principles for a TypeScript/Node.js project where most code is written by AI

> Date: 2026-05-18  
> Project: Telegram Mini App beauty marketplace  
> Context: ~90% of code will be generated by Claude, GPT-4 and similar LLMs  
> **Note:** Parts 9–11 contain project-specific examples. For the authoritative project  
> structure, file templates, and path aliases — see [Architecture.md](Architecture.md) §11–12.

---

## Why This Document Exists

When AI writes most of your code, the codebase is no longer written *for human readers first*. It is written so that an LLM can:

1. Find the right file on the first try (no wandering through 20 unrelated files)
2. Understand what a file does without reading the entire codebase
3. Add a new feature by following an existing pattern, not inventing a new one
4. Modify one feature without accidentally breaking another
5. Fit the relevant context into its context window (128k–200k tokens is not infinite)

Research shows that after restructuring a codebase for AI navigation, these metrics improved:
- AI finds the correct file on first try: ~60% → ~95%
- Generated code follows project patterns: ~40% → ~85%
- Time from prompt to working code: 8–12 min → 2–4 min

---

## Part 1: The Fundamental Problem — Context Windows

Every LLM has a context window. Claude Sonnet: ~200k tokens. GPT-4o: ~128k tokens. A token ≈ 0.75 words.

A large file (500+ lines) consumes the window. A deeply nested layered project forces loading 6+ files just to understand one feature. This means:

- **Large files** = LLM sees only part of the relevant code
- **Layered architecture** = adding a feature requires touching files in controllers/, services/, repositories/, models/, routes/ — all must be in context simultaneously
- **Barrel exports** (`export * from`) = LLM cannot tell which file a symbol comes from
- **Implicit dependencies** = LLM doesn't know what a function depends on without tracing 3 levels deep

The solution to all four problems is the same: **vertical slices with small, explicit, self-contained files**.

## Part 2: Vertical Slices vs. Horizontal Layers

### Horizontal Layers (traditional, bad for AI)

```
src/
├── controllers/
│   ├── masterController.ts     # knows about masters
│   ├── bookingController.ts    # knows about bookings
│   └── serviceController.ts   # knows about services
├── services/
│   ├── masterService.ts
│   ├── bookingService.ts
│   └── serviceService.ts      # naming already becomes absurd
├── repositories/
│   ├── masterRepository.ts
│   └── bookingRepository.ts
├── models/
│   ├── Master.ts
│   └── Booking.ts
└── routes/
    └── index.ts
```

**Problem:** To add the feature "master can update their own working hours", you must touch:
- `routes/index.ts`
- `controllers/masterController.ts`
- `services/masterService.ts`
- `repositories/masterRepository.ts`
- `models/Master.ts`

That is 5 files in 5 different folders. To fit them all in context, the LLM must see ~500+ lines of code unrelated to working hours (all the other features mixed in). It will make mistakes.

### Vertical Slices (feature-based, good for AI)

```
src/
├── features/
│   ├── master-profile/
│   │   ├── update-working-hours/
│   │   │   ├── update-working-hours.handler.ts   # 40 lines
│   │   │   ├── update-working-hours.schema.ts    # 20 lines
│   │   │   ├── update-working-hours.route.ts     # 15 lines
│   │   │   └── update-working-hours.test.ts      # 50 lines
│   │   └── get-master-profile/
│   │       └── ...
│   ├── booking/
│   │   ├── create-booking/
│   │   └── cancel-booking/
│   └── catalog/
│       ├── list-masters/
│       └── search-masters/
└── shared/
    ├── db/
    ├── auth/
    └── types/
```

**Result:** To add "master updates working hours", the LLM touches only one folder: `master-profile/update-working-hours/`. All 4 files fit in context simultaneously and contain nothing unrelated to working hours.

### The Core Principle

> Minimize coupling *between* slices. Maximize coupling *within* a slice.

Each slice should be as self-contained as possible. Different slices can use different approaches, different validators, even different database queries — and that is fine, because they never need to change together.

## Part 3: Feature-Sliced Design (FSD)

Feature-Sliced Design is a formal methodology for this approach. It originated in the frontend world but applies equally to backend Node.js projects.

### The 6 Layers (top to bottom)

```
app/          ← entry points, routing, server bootstrap, global middleware
pages/        ← route handlers assembled from lower layers (backend: HTTP routes)
widgets/      ← large self-contained feature groups (e.g., "entire booking flow")
features/     ← single user-facing operations with business value
entities/     ← business objects: Master, Booking, Service, Client
shared/       ← pure utilities, database client, config, types — no business logic
```

### The One Iron Rule

**A module can only import from layers below it. Never from layers at the same level or above.**

```
✅  features/booking/create-booking  →  imports from  entities/booking
✅  entities/booking                 →  imports from  shared/db
✅  features/catalog/list-masters    →  imports from  entities/master

✗   entities/booking                 →  imports from  features/booking   (upward)
✗   features/booking                 →  imports from  features/catalog   (same layer)
✗   shared/db                        →  imports from  entities/master    (upward)
```

This rule is what makes the architecture safe for AI code generation. An LLM cannot accidentally create circular dependencies, because the rule is mechanical and can be enforced with a linter (`eslint-plugin-boundaries` or `dependency-cruiser`).

### Anatomy of a Slice

Each slice inside `features/` follows the same internal structure:

```
features/
└── booking/
    └── create-booking/
        ├── create-booking.route.ts     # HTTP: registers the route
        ├── create-booking.schema.ts    # Zod: request/response validation
        ├── create-booking.handler.ts   # business logic: the actual work
        ├── create-booking.types.ts     # TypeScript types for this slice
        └── create-booking.test.ts      # tests co-located with the code
```

Every new feature the LLM creates follows this exact template. No guessing where things go.

### Segments (internal folders within a slice)

For larger slices, use named sub-folders:

| Segment | Contents |
|---------|----------|
| `ui/`   | React components, formatters (frontend) |
| `api/`  | HTTP request functions, response mappers |
| `model/`| Zod schemas, TypeScript interfaces, store |
| `lib/`  | Internal helpers specific to this slice |
| `config/`| Feature flags, constants |

## Part 4: Naming Conventions

Naming is the most important AI-friendliness factor after structure. Good names let the LLM predict file contents *before* opening the file.

### File Naming: kebab-case + type suffix

```
# Pattern: [domain]-[action].[type].ts

create-booking.handler.ts      # business logic function
create-booking.schema.ts       # Zod validation schema
create-booking.route.ts        # Fastify/Express route registration
create-booking.types.ts        # TypeScript type definitions
create-booking.test.ts         # test file
master-profile.entity.ts       # database entity / Prisma model wrapper
booking.repository.ts          # data access layer
auth.middleware.ts             # Fastify middleware
telegram-init-data.guard.ts    # input validation guard
```

**Why this matters for AI:** When the LLM sees `create-booking.handler.ts`, it knows before reading:
- This file contains the business logic for creating a booking
- It does NOT contain the route definition or schema
- When asked to "add validation", it should create or edit `create-booking.schema.ts`, not this file

### Variable and Function Naming

```typescript
// ✅ Explicit, no abbreviations
async function createBookingForMaster(
  masterId: string,
  clientTelegramId: number,
  serviceId: string,
  scheduledAt: Date
): Promise<CreateBookingResult>

// ✗ Abbreviated, context-dependent
async function create(m: string, c: number, s: string, d: Date)
```

```typescript
// ✅ Type names describe the shape exactly
type CreateBookingInput = {
  masterId: string;
  clientTelegramId: number;
  serviceId: string;
  scheduledAt: Date;
};

type CreateBookingResult =
  | { success: true; booking: Booking }
  | { success: false; reason: 'slot_taken' | 'master_not_found' | 'service_not_found' };

// ✗ Vague names
type Input = { ... }
type Result = { ... }
```

### Folder Naming Conventions

```
features/booking/create-booking/      # verb-noun: action on domain object
features/catalog/list-masters/        # verb-noun
features/master-profile/              # noun: domain area
entities/booking/                     # noun: business entity
shared/auth/                          # noun: cross-cutting concern
```

## Part 5: File Size and Complexity Limits

These are the practical limits for AI-friendly code. They are not absolute rules but heuristics that keep files within one logical concept.

| File type | Recommended max | Hard limit |
|-----------|----------------|------------|
| Handler (business logic) | 80 lines | 150 lines |
| Schema (Zod) | 50 lines | 100 lines |
| Route file | 30 lines | 60 lines |
| Entity definition | 60 lines | 120 lines |
| Test file | 100 lines | 200 lines |
| Shared utility | 50 lines | 100 lines |

**Why these limits exist for AI:**  
A 300-line handler file contains multiple responsibilities. The LLM must read all 300 lines to understand which part to modify. It may edit the wrong section or introduce conflicts. A 80-line handler does one thing. The LLM can read it in one pass and modify it confidently.

**Practical rule:** If you are about to write a function over 30 lines, ask whether it should be two functions. If a file is over 150 lines, ask whether it covers two concerns.

### Function Complexity

```typescript
// ✅ One function = one purpose
async function checkSlotAvailability(
  masterId: string,
  scheduledAt: Date
): Promise<boolean> {
  const conflictingBooking = await db.booking.findFirst({
    where: {
      masterId,
      scheduledAt,
      status: { not: 'cancelled' },
    },
  });
  return conflictingBooking === null;
}

async function createBookingRecord(
  input: CreateBookingInput
): Promise<Booking> {
  return db.booking.create({ data: input });
}

// Both are called by the handler, which orchestrates them
async function createBookingHandler(
  input: CreateBookingInput
): Promise<CreateBookingResult> {
  const isAvailable = await checkSlotAvailability(input.masterId, input.scheduledAt);
  if (!isAvailable) {
    return { success: false, reason: 'slot_taken' };
  }
  const booking = await createBookingRecord(input);
  return { success: true, booking };
}
```

The LLM can now be told: "modify `checkSlotAvailability` to also check the master's schedule settings" — and it knows exactly which file and function to touch.

## Part 6: Explicit Boundaries and Module Public APIs

### The Problem with Implicit Access

If every file can import from every other file directly, the LLM does not know what is "public" and what is an internal implementation detail. It will import internal functions from the wrong layer.

### Solution: Public API via index.ts

Each slice/entity exposes only what it intends to be public:

```typescript
// entities/booking/index.ts  ← PUBLIC API of the booking entity
export type { Booking, BookingStatus } from './booking.types';
export { BookingRepository } from './booking.repository';
export { BOOKING_STATUS } from './booking.constants';

// Internal files (NOT exported):
// booking.db-mapper.ts — internal, not exported
// booking.sql-helpers.ts — internal, not exported
```

```typescript
// ✅ LLM imports from the public API
import { Booking, BookingRepository } from '@/entities/booking';

// ✗ LLM should never import internal files directly
import { someHelper } from '@/entities/booking/booking.sql-helpers';
```

**Important:** Do NOT use `export * from './booking.types'` (wildcard re-exports). Use explicit named exports. Wildcards hide the public API and make it impossible to know what is available without reading every file.

### Path Aliases

Set up TypeScript path aliases so the LLM uses consistent import paths:

```json
// tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "@/domain/*":         ["src/domain/*"],
      "@/use-cases/*":      ["src/use-cases/*"],
      "@/adapters/*":       ["src/adapters/*"],
      "@/infrastructure/*": ["src/infrastructure/*"],
      "@/shared/*":         ["src/shared/*"]
    }
  }
}
```

The LLM then always writes `import { X } from '@/domain/booking'`, never `import { X } from '../../../domain/booking'`. Relative paths are context-dependent and confusing.

The aliases mirror Clean Architecture layers — an import path tells you immediately which layer a symbol belongs to.

## Part 7: Dependency Injection vs Direct Imports

This is a common debate. For AI-written code, the answer is practical:

### Use Direct Imports for Stateless Utilities

```typescript
// ✅ Direct import — no side effects, pure function
import { formatPrice } from '@/shared/formatters';
import { BookingSchema } from './create-booking.schema';
```

### Use Dependency Injection for Stateful Services (Database, External APIs)

```typescript
// ✅ Dependencies visible in the function signature
// The LLM can read the signature and know exactly what this function needs

async function createBookingHandler(
  input: CreateBookingInput,
  deps: {
    bookingRepo: BookingRepository;
    masterRepo: MasterRepository;
    notificationService: NotificationService;
  }
): Promise<CreateBookingResult> {
  // ...
}
```

**Why DI matters for AI:**  
When a function takes its dependencies as parameters, the LLM can see from the signature alone what the function needs. It will not accidentally import a database connection inside a business logic function. It will not create hard-to-test code. The function signature is self-documenting.

**Why full DI containers are bad for AI:**  
Complex DI frameworks (InversifyJS, TSyringe) add decorators, tokens, and magic that the LLM struggles to replicate correctly. Prefer simple constructor injection or parameter injection for small projects.

### Practical Pattern: Repository Abstraction (Port/Adapter)

```typescript
// domain/ports/booking.repo.port.ts — Port interface (the LLM programs against this)
export interface IBookingRepository {
  findById(id: string): Promise<Booking | null>;
  create(input: CreateBookingInput): Promise<Booking>;
  cancel(id: string, by: 'client' | 'master' | 'system'): Promise<void>;
}

// adapters/repositories/postgres-booking.repo.ts — Adapter implementation (raw SQL)
export class PostgresBookingRepository implements IBookingRepository {
  constructor(private readonly pool: Pool) {}

  async findById(id: string): Promise<Booking | null> {
    const { rows } = await this.pool.query(
      'SELECT * FROM bookings WHERE id = $1', [id]
    );
    return rows[0] ?? null;
  }

  async create(input: CreateBookingInput): Promise<Booking> {
    const { rows } = await this.pool.query(
      `INSERT INTO bookings (client_id, master_id, service_id, start_time, end_time, price_snapshot)
       VALUES ($1,$2,$3,$4,$5,$6) RETURNING *`,
      [input.clientId, input.masterId, input.serviceId, input.startTime, input.endTime, input.price]
    );
    return rows[0];
  }

  async cancel(id: string, by: 'client' | 'master' | 'system'): Promise<void> {
    await this.pool.query(
      `UPDATE bookings SET status='CANCELLED', cancelled_by=$1, cancelled_at=now()
       WHERE id=$2`,
      [by, id]
    );
  }
}
```

The LLM writes Use Cases against `IBookingRepository`. Changing the database touches only `PostgresBookingRepository`, not a single Use Case.

## Part 8: Adding a New Feature Without Breaking Existing Ones

This is the Open/Closed Principle applied for AI development.

### The Pattern: Additive Changes Only

AI should be able to add a new feature by:
1. Creating a new folder `features/X/new-feature/`
2. Creating 3–4 new files inside it
3. Registering the route in one central place

It should NOT need to:
- Modify existing handler files
- Add branches to existing service functions
- Change shared database queries

### Concrete Example: Adding "Get Available Slots" Feature

```
Before adding the feature, the project has:
features/booking/create-booking/
features/booking/cancel-booking/

After adding:
features/booking/create-booking/       ← UNTOUCHED
features/booking/cancel-booking/       ← UNTOUCHED
features/booking/get-available-slots/  ← NEW
    ├── get-available-slots.route.ts
    ├── get-available-slots.schema.ts
    ├── get-available-slots.handler.ts
    └── get-available-slots.test.ts

app/routes.ts — ONE LINE ADDED to register the new route
```

Zero risk of breaking `create-booking` or `cancel-booking` because you never touched them.

### Route Registration Pattern

```typescript
// app/routes.ts — the ONLY file that registers routes
import Fastify from 'fastify';
import { createBookingRoute } from '@/features/booking/create-booking/create-booking.route';
import { cancelBookingRoute } from '@/features/booking/cancel-booking/cancel-booking.route';
import { getAvailableSlotsRoute } from '@/features/booking/get-available-slots/get-available-slots.route'; // ← new line only

export function registerRoutes(app: ReturnType<typeof Fastify>) {
  createBookingRoute(app);
  cancelBookingRoute(app);
  getAvailableSlotsRoute(app); // ← new line only
}
```

The LLM adds 2 lines to one file and creates 4 new files. It cannot accidentally break existing routes.

## Part 9: CLAUDE.md — Persistent AI Context

`CLAUDE.md` is a markdown file at the project root that Claude Code reads automatically at the start of every session. It is the single most impactful AI-friendliness improvement you can make.

### What to Include

Key sections every CLAUDE.md should have:

```markdown
# Project: [Name]

## Stack
One line per technology. Include versions. No marketing text.

## Architecture
State the pattern. State the Dependency Rule. State where each type of code lives.

## Naming conventions
File patterns, function naming, type naming. AI uses these as defaults.

## Forbidden patterns ("Never" list)
The most important section. What the AI must NOT do, with reasons.
Examples:
- Never write SQL outside adapters/repositories/
- Never import from infrastructure/ in domain/ or use-cases/
- Never use window.Telegram.WebApp directly — wrap in shared/lib/telegram.ts

## Commands
npm run dev / test / lint / db:migrate

## Key files
Point to the architecture doc, the schema file, the entry point.
```

For this project, the CLAUDE.md is at [CLAUDE.md](../CLAUDE.md). It covers Clean Architecture layers, path aliases, security rules, and the full "Never" list.

### What NOT to Include in CLAUDE.md

- Commands already obvious from `package.json`
- Patterns that Claude already follows naturally
- More than ~150 lines (quality degrades past that)
- Verbose inline documentation (reference files with paths instead)
- Code already visible in the source tree — don't duplicate it in CLAUDE.md

## Part 10: Project Structure

The principles in this document lead to a Clean Architecture layout. The authoritative structure for this project is documented in [Architecture.md §11](Architecture.md#11-структура-папок).

### Backend — Clean Architecture layers

```
src/
├── domain/          ← CORE: entities, value objects, port interfaces, domain services
│   ├── booking/     ← booking.entity.ts, booking.events.ts
│   ├── master/      ← master.entity.ts
│   ├── services/    ← slot-calculator.ts (pure function, no I/O)
│   └── ports/       ← IBookingRepository, INotificationPort, IEventBus, IFileStoragePort
│
├── use-cases/       ← ORCHESTRATION: use ports, no SQL, no HTTP
│   ├── create-booking/create-booking.use-case.ts
│   ├── cancel-booking/cancel-booking.use-case.ts
│   └── get-available-slots/get-available-slots.use-case.ts
│
├── adapters/        ← FORMAT TRANSLATION
│   ├── http/        ← Fastify controllers + routes + Zod schemas
│   └── repositories/← postgres-booking.repo.ts implements IBookingRepository
│
└── infrastructure/  ← EXTERNAL SYSTEMS: pg Pool, Redis, Grammy, S3, BullMQ
    ├── postgres/
    ├── redis/
    ├── telegram/    ← notification-adapter.ts implements INotificationPort
    ├── queue/
    └── event-handlers/
```

### Dependency direction

```
domain ← use-cases ← adapters ← infrastructure
```

Only inward. Never outward. `domain/` knows nothing about Fastify, PostgreSQL, or BullMQ.

### Frontend — Feature-Sliced Design

The frontend follows FSD (4 layers). See [Architecture.md §11](Architecture.md#11-структура-папок) for the full frontend structure with all pages, features, entities, and shared layers.

## Part 11: Concrete File Examples

For project-specific templates (Use Case, Schema, Controller, Routes) see [Architecture.md §12](Architecture.md#12-шаблоны-файлов-для-ai-разработки).

The patterns below show the universal principles that apply regardless of project specifics.

### Use Case File Template (universal pattern)

```typescript
// use-cases/[verb-noun]/[verb-noun].use-case.ts
// Depends only on Port interfaces from domain/ — no SQL, no HTTP, no framework imports

export type [VerbNoun]Input = { /* primitive types only */ };
export type [VerbNoun]Result =
  | { ok: true;  data: ResultType }
  | { ok: false; reason: 'reason_a' | 'reason_b' };  // explicit union, not throw

export class [VerbNoun]UseCase {
  constructor(
    private readonly repo: IEntityRepository,   // interface, not concrete class
    private readonly eventBus: IEventBus,
  ) {}

  async execute(input: [VerbNoun]Input): Promise<[VerbNoun]Result> {
    // 1. Validate business rules (not HTTP validation — that's in schema)
    // 2. Call repos via interfaces
    // 3. Publish domain events
    // 4. Return result — never throw for expected outcomes
  }
}
```

### Schema File Template (universal pattern)

```typescript
// adapters/http/[noun]/[verb-noun].schema.ts
// Validates HTTP request body — lives in adapters layer, not domain
import { z } from 'zod';

export const [VerbNoun]BodySchema = z.object({
  id:        z.string().uuid(),
  timestamp: z.string().datetime().transform((s) => new Date(s)),
  // ... other fields
});

export type [VerbNoun]Body = z.infer<typeof [VerbNoun]BodySchema>;
```

### Controller + Routes File Template (universal pattern)

```typescript
// adapters/http/[noun]/[noun].controller.ts
// Translates HTTP ↔ Use Case. No business logic.
import type { FastifyRequest, FastifyReply } from 'fastify';
import { [VerbNoun]UseCase } from '@/use-cases/[verb-noun]/[verb-noun].use-case';
import { Postgres[Entity]Repository } from '@/adapters/repositories/postgres-[entity].repo';
import { pool } from '@/infrastructure/postgres/pool';
import { eventBus } from '@/infrastructure/event-bus';

export async function [verbNoun]Controller(req: FastifyRequest, reply: FastifyReply) {
  const useCase = new [VerbNoun]UseCase(new Postgres[Entity]Repository(pool), eventBus);
  const result = await useCase.execute({ ...req.body, userId: req.user.userId });

  if (!result.ok) return reply.status(400).send({ error: result.reason });
  return reply.status(201).send(result.data);
}

// adapters/http/[noun]/[noun].routes.ts
export async function [noun]Routes(app: FastifyInstance) {
  app.post('/api/v1/[noun]', {
    preHandler: [app.authenticate],
    schema: { body: [VerbNoun]BodySchema },
    handler: [verbNoun]Controller,
  });
}
```

## Part 12: Summary of Principles

| Principle | What it Solves | Practical Rule |
|-----------|---------------|----------------|
| **Vertical slices** | LLM touches too many files for one feature | One folder per use case, all related files inside |
| **Small files** | LLM cannot fit the whole file in context | Max 150 lines per file, split if needed |
| **Explicit naming** | LLM cannot predict file contents | `[verb-noun].[type].ts` pattern everywhere |
| **One concept per file** | LLM edits the wrong section | If a file has two responsibilities, split it |
| **Explicit public APIs** | LLM imports internal implementation details | Use `index.ts` with explicit named exports |
| **Dependency injection** | LLM cannot understand hidden dependencies | Make deps visible in function signatures |
| **No cross-layer violations** | LLM accidentally breaks Dependency Rule | domain/ never imports from infrastructure/; enforced by linter |
| **Co-located tests** | LLM forgets to update tests | Test file lives next to the source file |
| **CLAUDE.md** | LLM re-invents patterns every session | Write down the rules once, they persist across sessions |
| **Path aliases** | LLM uses unstable relative paths | `@/domain/`, `@/use-cases/`, `@/adapters/`, `@/infrastructure/`, `@/shared/` everywhere |
| **Additive changes** | LLM modifies existing code unnecessarily | New feature = new folder, not modification of existing handler |
| **Unidirectional dependencies** | LLM creates circular imports | Enforce with `dependency-cruiser` in CI |
