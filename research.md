## 5 Примеров Telegram Mini App в нише бьюти и похожих нишах

---

### 1. MastApp (mastapp.ru)

**Что делает:**  
Telegram Mini App для частных мастеров (ногти, ресницы, массаж, тренеры). Клиент открывает бота по ссылке из Instagram/WhatsApp — выбирает услугу, дату, время без установки приложений. Мастер получает уведомление о каждой записи. 2 500+ мастеров активно используют, 199 ₽/мес.

**Сильные стороны:**
- Нулевой барьер для клиента — всё внутри Telegram, который уже стоит
- Автоматические напоминания клиенту перед приёмом
- Поддержка предоплаты / залога для снижения no-show
- Ссылка для любого канала (Instagram, WhatsApp, ВКонтакте)
- Сообщается о росте записей на 40–60% после перехода

**Слабые стороны:**
- Нет портфолио и фото-галереи работ мастера
- Минимальная кастомизация — нельзя сделать «своё» приложение визуально
- Нет CRM (история клиента, заметки, теги)
- Нет системы отзывов и рейтингов

**Что можно сделать лучше:**
- Добавить галерею работ прямо в карточку мастера
- Дать возможность кастомизации цвета и логотипа
- Встроить историю визитов для клиента

### 2. Slotly (slotlycrm.ru) — для мастеров маникюра

**Что делает:**  
CRM + Mini App специально для ногтевых мастеров и маникюрных студий. Клиент выбирает тип услуги (классика, аппаратный, гель, дизайн), видит свободные слоты, подтверждает запись. Есть фото-каталог дизайнов ногтей. Работает в Telegram и VK MAX. Старт от 490 ₽/мес, 60 дней бесплатно.

**Сильные стороны:**
- **Фото-каталог дизайнов ногтей** — клиент выбирает и сразу показывает мастеру что хочет
- Расчёт времени для нескольких услуг за один визит
- Соответствие 54-ФЗ (онлайн-касса для официальных мастеров)
- True Mini App с графическим UI (не просто текстовые кнопки в чате)

**Слабые стороны:**
- Нет портфолио конкретного мастера (биография, фото «до/после», отзывы о мастере)
- Нет программы лояльности и скидочной механики
- На начальном тарифе — только один мастер
- Ориентация только на российский рынок

**Что можно сделать лучше:**
- Совместить каталог дизайнов с профилем мастера
- Добавить раздел «мои мастера» — чтобы клиент возвращался к своему
- Дать возможность клиенту загрузить референс-фото

### 3. MyAppoint (Telegram-бот @MyAppointAdminBot)

**Что делает:**  
Бесплатная альтернатива YClients. Клиент выбирает мастера → услугу → дату → время без регистрации. У мастера — личная веб-страница для ссылки в шапке Instagram. Встраиваемый виджет, QR-код для офлайн-промо, AI-ассистент на естественном языке («покажи запись на завтра», «перенеси Аню на 15:00»). Поддержка групповых сессий (йога, мастер-классы).

**Сильные стороны:**
- AI-управление расписанием на естественном языке — реально полезно для занятых мастеров
- Несколько точек входа: бот + веб + виджет + QR — охватывает все каналы привлечения
- Запись без регистрации клиента снижает drop-off
- Веб-панель с календарём, базой клиентов, CSV-экспортом, статистикой загрузки

**Слабые стороны:**
- Нет фото-портфолио работ мастера
- Групповые сессии избыточны для nail/lash/hair мастеров
- AI может неверно интерпретировать запрос
- Перегруженный функционал для очень маленьких операторов

**Что можно сделать лучше:**
- Сфокусироваться на конкретных нишах (ногти/ресницы/брови) с отраслевыми функциями
- Сделать более простой и быстрый онбординг для мастера

### 4. Salon Booking Bot (open-source, GitHub: nazgool97)

**Что делает:**  
Самохостинговая open-source система бронирования для салонов. Сочетает классический Telegram-бот и Mini App с мобильным UI. Поддерживает роли: Клиент / Мастер / Владелец. Панель аналитики (LTV, Retention Rate, No-Show rate), конкурентная безопасность через PostgreSQL Advisory Locks.

**Сильные стороны:**
- Двойной интерфейс: Mini App + классические кнопки бота — охватывает разные уровни tech-грамотности клиентов
- Аналитика для владельца на уровне продакшна (LTV, no-show rate)
- Полный контроль данных, нет SaaS-подписки
- Конкурентная обработка бронирований (нет двойной записи под нагрузкой)

**Слабые стороны:**
- Нет портфолио/галереи для мастеров
- Нет системы отзывов и рейтингов
- Требует технических знаний для деплоя
- Нет онлайн-оплаты из коробки, нет программы лояльности

**Что можно сделать лучше:**
- Добавить галерею работ и отзывы как core-фичи
- Дать готовый hosted-вариант без необходимости деплоя

### 5. SmartMove Realty (Telegram Mini App для недвижимости)

**Что делает:**  
TMA агентства недвижимости: листинг объектов в виде карточек с фото, фильтры, избранное, запись на просмотр по календарю, уведомления о снижении цены. Ниша другая, но паттерн «каталог → карточка → запись» — идентичен бьюти.

**Сильные стороны:**
- Каталог карточек + фильтры + детальная страница + бронирование — готовая UX-архитектура для бьюти
- Сохранённые «Избранное» через Telegram user ID — клиент возвращается к своему мастеру
- Push-уведомления о «новом слоте» (аналог уведомления о снижении цены) возвращают пользователей
- Bounce rate снизился с 72% до 18% — Telegram-контекст держит пользователей

**Слабые стороны:**
- Нет системы отзывов и рейтингов — критично для высокодоверительных сервисов
- Нет программы лояльности

**Что можно сделать лучше:**
- Для бьюти: заменить «цену за квартиру» на «цену + длительность» прямо на карточке
- Добавить «Избранные мастера» как ключевую фичу retention
- Уведомления «появился слот у Маши» — прямой аналог «снижение цены»

## Что мы можем сделать лучше всех конкурентов

| Разрыв в конкурентах | Наше решение |
|---|---|
| Нет портфолио мастера (у всех) | Галерея работ в профиле мастера: фото до/после, стили |
| Нет системы отзывов (почти ни у кого) | Рейтинг + текстовые отзывы с верификацией через Telegram |
| Нет «избранных мастеров» | Сохранить мастера → один тап для повторной записи |
| Нет каталога дизайнов/референсов | Раздел «Вдохновение» — клиент выбирает стиль ДО выбора мастера |
| Нет программы лояльности (почти) | Баллы за визиты, уровни клиента, реферальная механика |
| Запись только вперёд, нет рескедьюлинга | Кнопка «Перенести» прямо в карточке записи |
| Нет ощущения «своего» приложения | Кастомизация: имя мастера, фото, цвет бренда |

## Навигация: что работает в Telegram Mini Apps

**Рекомендуемый паттерн: Bottom Tab Bar + Stack Navigation**

```
Нижняя панель (4 вкладки):
[Каталог]  [Мои записи]  [Избранное]  [Профиль]

Стек внутри вкладки "Каталог":
Главная → Категория → Услуга → Мастер → Дата/Время → Итог → Оплата → Успех
```

**Правила навигации:**
- `WebApp.BackButton` — системная кнопка «Назад» в заголовке Telegram. Не создавать кастомную внутри UI.
- `WebApp.MainButton` — единственный главный CTA на экране. Текст меняется динамически: «Выбрать время» → «Подтвердить запись» → «Оплатить ₽X»
- Скрывать Tab Bar во время активного booking flow (шаги 3–7)
- Максимум 4 уровня вложенности в стеке
- Никаких горизонтальных каруселей у краёв экрана на iOS — конфликт со свайпом для закрытия

## Telegram Mini App API: что использовать

| API | Применение |
|---|---|
| `WebApp.ready()` | Сразу при монтировании — убирает нативный loader |
| `WebApp.themeParams` | Синхронизация цветов с темой Telegram (обязательно) |
| `WebApp.MainButton` | Главный CTA, меняет текст по шагам, `showProgress()` при запросе |
| `WebApp.BackButton` | Показывать не на корневых экранах, скрывать на корне |
| `WebApp.HapticFeedback` | `impactOccurred('light')` на тапы; `notificationOccurred('success')` при подтверждении |
| `WebApp.initDataUnsafe.user` | Персонализация приветствия, авто-заполнение имени |
| `WebApp.enableClosingConfirmation()` | На экране итога и оплаты |
| `WebApp.requestContact()` | Телефон — только после выбора услуги и слота, не при старте |
| `WebApp.openInvoice()` | Нативный Telegram Payment |
| `WebApp.shareToStory()` | После успешной записи: «Поделиться в Stories» |
| `WebApp.CloudStorage` | Кеш предпочтений, последние просмотренные услуги |
| `WebApp.expand()` | При запуске — развернуть на весь экран сразу |

## Что заставляет пользователей записаться (конверсия)

| Фактор | Эффект |
|---|---|
| Цена видна на карточке каталога | Убирает «сюрпризы» на этапе оплаты |
| «3 слота доступно сегодня» на карточке | +32% к переходам в booking flow |
| Выбор мастера ДО выбора даты | Выше completion — пользователь «привязывается» к своему мастеру |
| Не более 4 шагов до подтверждения | Каждый лишний экран = −20% завершений |
| Telegram Payments (нативно) | Убирает вопрос «безопасно ли?» — карта уже сохранена в Telegram |
| Напоминание за 48ч и 2ч | Снижение no-show на 50–60% |
| «Добавить в календарь» после записи | No-show в 6x ниже у тех, кто добавил |
| Персонализированное приветствие | Выше depth сессии |

## Что заставляет пользователей уходить (bounce)

| Фактор | Эффект |
|---|---|
| Регистрация через email до просмотра каталога | 60–70% немедленного drop-off |
| Цена скрыта до экрана оплаты | Главная причина брошенных корзин |
| Загрузка >3 сек | −40% пользователей не дожидаются |
| Редирект на внешний сайт для оплаты | ~30% не возвращаются |
| Нет политики отмены до подтверждения | Тревога, споры, плохие отзывы |
| Нет кнопки «Назад» или она сломана | Rage-quit |
| Нет подтверждения / экрана успеха | Пользователь не уверен, записался ли он |
| Приложение выглядит как сайт в iframe | Потеря доверия |

## Модель продукта: Маркетплейс для мастеров

> Уточнение концепции: одно приложение для всех мастеров. Каждый мастер регистрируется, создаёт профиль и работает с клиентами через личный кабинет. Клиент видит каталог всех мастеров и записывается к нужному.

Это **маркетплейс**, а не инструмент для одного мастера. Меняет архитектуру, UX и бизнес-модель.

---

### Что меняется в UX по сравнению с одиночным инструментом

| Аспект | Одиночный инструмент | Маркетплейс |
|---|---|---|
| Главный экран | Каталог услуг одного мастера | Каталог мастеров + фильтры |
| Единица контента | Услуга | Мастер (у каждого свои услуги) |
| Первый шаг клиента | Выбрать услугу | Найти мастера или выбрать специализацию |
| Навигация | Линейная | Двойная: мастера → услуги конкретного мастера |
| Доверие | Один бренд | Нужны рейтинги и отзывы для каждого мастера |
| Бэкенд | Простой | Мультитенантный |

---

### Роли в системе

**Клиент**
- Просматривает каталог мастеров
- Фильтрует по специализации, цене, рейтингу, доступности
- Смотрит профиль мастера: портфолио, услуги, цены, отзывы
- Записывается и управляет записями

**Мастер**
- Регистрируется на платформе через бота
- Заполняет профиль: фото, специализация, описание, галерея работ
- Создаёт список услуг с ценами и длительностью
- Настраивает расписание (рабочие дни, часы, перерывы)
- Видит свои записи в личном кабинете
- Получает уведомления о новых записях и отменах

**Администратор платформы (владелец)**
- Модерирует профили мастеров
- Видит аналитику по всей платформе
- Управляет категориями и тегами
- Настраивает комиссию или тарифы

| Экран | Назначение |

| **Мои мастера / Избранное** | Сохранённые мастера для быстрого повторного обращения |

| Экран | Назначение |

| **Расписание** | Календарь с записями, управление рабочим временем |
| **Галерея работ** | Загрузка фото, подписи, категории стилей |
| **Мои клиенты** | История визитов, контакты, заметки |
| **Уведомления** | Новые записи, отмены, напоминания |
| **Настройки профиля** | Фото, описание, контакты, соцсети |

### Архитектурные решения для маркетплейса

**База данных — мультитенантность через `master_id`**

Все данные привязаны к мастеру. Клиент — отдельная сущность, не привязанная к мастеру до момента записи.

```
masters          — профиль мастера (tg_id, name, bio, photo, category)
services         — услуги мастера (master_id, name, price, duration, photo)
schedules        — расписание мастера (master_id, weekday, time_from, time_to)
bookings         — записи (client_tg_id, master_id, service_id, datetime, status)
reviews          — отзывы (client_tg_id, master_id, booking_id, rating, text)
master_gallery   — фото работ (master_id, photo_url, caption)
```

**Два режима входа в одном боте**

Telegram-бот определяет роль по `tg_id` при старте:
- Если пользователь зарегистрирован как мастер → открывается Mini App в режиме «Личный кабинет»
- Если нет → открывается клиентский каталог
- Переключение через меню или команду `/switch`

**Регистрация мастера**  
Не через Mini App, а через обычный бот-диалог (проще реализовать онбординг с шагами). После регистрации — Mini App с личным кабинетом.

**Deeplink на конкретного мастера**  
`t.me/YourBot?startapp=master_123` — каждый мастер получает личную ссылку для размещения в Instagram, ВКонтакте. По ссылке клиент попадает сразу на профиль этого мастера, минуя общий каталог.

### Бизнес-модели монетизации платформы

| Модель | Описание | Плюсы | Минусы |
|---|---|---|---|
| **Подписка мастера** | Фиксированная плата (например, 299–999 ₽/мес) | Предсказуемый доход | Мастера уходят в off-season |
| **Комиссия с записи** | % от стоимости каждой записи | Масштабируется с платформой | Нужен приём платежей, сложнее |
| **Freemium** | Бесплатно до N записей/мес, платно сверху | Низкий барьер входа для мастера | Требует жёсткого контроля лимитов |
| **Витрина бесплатно, реклама платно** | Профиль бесплатен, поднятие в поиске — платно | Быстро набирает мастеров | Сложно объяснить ценность |

**Рекомендация для старта:** Freemium — бесплатно первые 30 записей/мес, затем подписка. Так набирается база мастеров быстро.

### Ключевые UX-риски маркетплейса

**1. Проблема «холодного старта»**  
Клиенты не придут, если нет мастеров. Мастера не придут, если нет клиентов.  
→ Решение: сначала привлечь 10–20 мастеров вручную, затем открывать для клиентов.

**2. Доверие к незнакомому мастеру**  
В одиночном инструменте клиент уже знает мастера из Instagram. В маркетплейсе — нет.  
→ Обязательны: рейтинг, отзывы, верифицированный профиль, фото до/после.

**3. Качество профилей мастеров**  
Мастера заполнят профиль плохо — без фото, с минимальным описанием.  
→ Решение: онбординг с прогресс-баром «Заполненность профиля», блокировка публикации при незаполненных обязательных полях.

**4. Конкуренция между мастерами одной специализации**  
Мастера видят конкурентов рядом с собой.  
→ Решение: сортировка по рейтингу и активности, а не по алфавиту. Хорошая работа = видимость.

### UX-дизайнер: ошибки новичков, что обязательно, что бесит

#### Типичные ошибки новичков

**1. Делают сайт внутри Telegram, а не Mini App**  
Копируют дизайн сайта в WebView: своя шапка, своя кнопка «Назад», своя цветовая схема. Выглядит как фишинг. Пользователь теряет доверие с первого экрана.  
→ Решение: использовать `WebApp.themeParams`, нативный `BackButton`, нативный `MainButton`.

**2. Регистрация в начале**  
«Введите имя, email, телефон» — до того как показали хоть одну услугу. Telegram уже знает, кто пользователь. Просить данные вручную — потерять 60–70% новых посетителей сразу.  
→ Решение: сразу открывать каталог. Телефон запрашивать через `requestContact()` только на шаге подтверждения.

**3. Плоский список услуг без визуала**  
Список из 15 позиций текстом без фото, без цены, без длительности на каждой строке. Клиент не понимает разницу между «маникюр классический» и «маникюр аппаратный».  
→ Решение: карточки с фото, ценой и длительностью прямо в каталоге.

**4. Слишком длинный booking flow**  
Больше 4 шагов до подтверждения — каждый дополнительный экран убирает ~20% пользователей. Новички добавляют экраны «на всякий случай».  
→ Решение: максимум 4 шага: Услуга → Мастер → Дата/Время → Итог.

**5. Нет экрана подтверждения**  
После нажатия «Записаться» — просто уходит обратно в каталог. Пользователь не знает, получилось ли. Перезаписывается второй раз.  
→ Решение: экран успеха с анимацией, деталями записи и кнопкой «Добавить в календарь».

**6. Горизонтальные карусели у краёв экрана на iOS**  
Swipe-карусель фото у левого или правого края → срабатывает системный жест закрытия Mini App. Пользователь вылетает из приложения в середине просмотра.  
→ Решение: отступ минимум 16–20px от краёв для любых горизонтальных элементов.

**7. Не вызывают `WebApp.expand()`**  
Mini App открывается в «полуэкранном» режиме — маленькая шторка снизу. Пользователь видит обрезанный контент и думает, что приложение сломано.  
→ Решение: `WebApp.expand()` при монтировании — всегда.

#### Что обязательно должно быть

- **Цена и длительность на каждой карточке** в каталоге — без клика для просмотра
- **Фото услуги или работ мастера** — бьюти — визуальная ниша, текст не продаёт
- **Индикатор доступности** — «доступно сегодня», «3 свободных слота» прямо на карточке
- **Sticky MainButton** — единственный CTA внизу, текст меняется по шагам
- **Экран успеха** с деталями и возможностью добавить в календарь
- **Политика отмены** видна ДО подтверждения — иначе споры и негативные отзывы
- **Скелетон-загрузка** вместо спиннера — воспринимается в 2–3 раза быстрее

#### Что бесит пользователей

- Цена появляется только на экране оплаты — чувствуют себя обманутыми
- Нет возможности перенести запись — только отменить и создать заново
- Сломанная кнопка «Назад» или её отсутствие — пользователь застрял
- Приложение не помнит, что пользователь делал в прошлый раз — надо всё начинать сначала
- Уведомления о записи приходят в отдельный канал, а не в диалог с ботом
- Медленная загрузка без скелетона — белый экран дольше 2 секунд = закрыл

### Маркетолог: что заставит записаться, элементы конверсии

#### Триггеры первого бронирования

**1. Социальное доказательство на первом экране**  
«Маша — 4.9 ★, 247 записей» прямо на карточке мастера. Рейтинг и количество клиентов убирают тревогу перед первым визитом сильнее, чем любой рекламный текст.

**2. Дефицит и срочность без манипуляций**  
«Осталось 2 слота на эту неделю» — правдивая информация о реальной занятости мастера. Работает лучше любых скидок, не разрушает ценность услуги.

**3. Цена видна сразу — без «уточняйте»**  
Мастера часто боятся показывать цены, думая, что отпугнут. На самом деле скрытая цена отпугивает сильнее. Клиент, видящий цену заранее, записывается и приходит. Клиент, узнающий цену на месте, — нет.

**4. Персонализация с первого экрана**  
«Привет, Алина — выбери свою услугу» (имя из Telegram). Конверсия в booking flow выше у персонализированных экранов — пользователь ощущает, что приложение «знает» его.

**5. Портфолио до кнопки «Записаться»**  
Клиент, посмотревший 3+ фото работ мастера, конвертируется в запись значительно чаще, чем тот, кто не смотрел. Фото работают как «примерка» — снижают страх получить не то.

#### Элементы повышения конверсии (приоритет по влиянию)

| Элемент | Почему работает |
|---|---|
| Рейтинг + число клиентов на карточке | Социальное доказательство снимает тревогу |
| «Доступно сегодня» / «3 слота» | Срочность без давления |
| Цена и длительность в каталоге | Предсказуемость = доверие |
| Фото работ (3–5 на мастера) | Визуальная «примерка» результата |
| Выбор мастера ДО даты | Клиент «привязывается» к мастеру и ищет подходящий слот |
| Telegram Payments нативно | Карта уже сохранена в TG — нет страха вводить данные |
| «Добавить в календарь» после записи | Снижает no-show в 6 раз |
| Напоминание за 48ч + 2ч | Снижает no-show на 50–60% |
| «Ваш прошлый мастер — Маша» при повторном визите | Возврат к проверенному мастеру в 1 тап |

#### Механики retention (повторных записей)

- **Напоминание через бота**: «Ваш маникюр — 3 недели назад. Пора обновить?» — срабатывает как личное сообщение, не как реклама
- **Избранные мастера**: клиент добавляет мастера → возвращается без поиска
- **История визитов**: «Вы были у Маши 4 раза» — создаёт лояльность через историю
- **Уведомление об открытии слота**: «Появилось окошко у Маши в пятницу в 14:00» — горячий лид

#### Что НЕ делать в маркетинге внутри приложения

- Поп-апы и баннеры при открытии — первое впечатление должно быть каталогом, не рекламой
- Скидки на первый визит на главном экране — обесценивает услугу до первого контакта
- Обязательная подписка на канал до просмотра каталога — барьер, не ценность

### Telegram-разработчик: ограничения, что работает, что не делать

#### Жёсткие ограничения платформы

**Размер и производительность**  
- Bundle JS: рекомендуется до 2 МБ. Больше — долгая загрузка, штраф в App Store Telegram
- Изображения: использовать WebP, lazy loading, CDN. Без оптимизации каталог с фото «убьёт» загрузку
- LCP (Largest Contentful Paint) должен быть ≤ 2.5 сек. Telegram не терпит медленных приложений

**Сессии и состояние**  
- Mini App закрывается, если пользователь переходит в другой чат. При возврате — новая сессия. Состояние хранить на бэкенде, не в памяти браузера
- `CloudStorage` API — 1024 ключа по 4096 символов. Можно хранить предпочтения и черновик записи

**Верификация данных**  
- `initDataUnsafe` на клиенте — только для UI. Всегда проверять `initData` HMAC-подпись на бэкенде. Иначе любой может назваться чужим Telegram ID

**Платежи**  
- Telegram Payments работает через платёжных провайдеров (ЮKassa, Stripe, Payme и др.). Нужен отдельный merchant account у провайдера. Прямого вывода нет — нужен счёт юрлица или самозанятость
- Telegram Stars — только для цифровых товаров, не подходит для записи к мастеру

**Хостинг Mini App**  
- HTTPS обязателен — Telegram не откроет HTTP
- Нет требований к серверу — работает любой хостинг (Vercel, Netlify, VPS). Для продакшна нужен бэкенд для хранения записей

#### Что работает хорошо

| Возможность | Применение |
|---|---|
| `initDataUnsafe.user` | Имя, username, ID — без регистрационных форм |
| `requestContact()` | Телефон одним тапом — 80%+ пользователей разрешают |
| `MainButton` | Нативный, быстрый, доверенный — конверсия выше кастомных кнопок |
| `HapticFeedback` | Тактильный отклик — ощущение нативного приложения |
| `themeParams` | Авто-поддержка светлой/тёмной темы без усилий |
| Bot messages | Подтверждения и напоминания в том же диалоге с ботом — не теряются |
| Deeplinks | `t.me/bot?startapp=service_123` — прямая ссылка на конкретную услугу из Instagram |
| `shareToStory` | Вирусное распространение после успешной записи |

#### Что лучше не делать

**Не использовать `window.history` для навигации**  
Telegram WebView обрабатывает историю браузера непредсказуемо на iOS и Android. Реализовывать навигацию через собственный стек состояний (React Router с `MemoryRouter`, Zustand, etc.).

**Не делать внешние редиректы в середине flow**  
Переход на внешний сайт (банк, форма оплаты, Instagram) прерывает сессию. 30% пользователей не возвращаются. Использовать `openInvoice()` для оплаты, `openTelegramLink()` для TG-ссылок.

**Не хранить booking-данные только в памяти React**  
Пользователь выбрал услугу и мастера → перешёл в другой чат → вернулся. Данные потеряны. Каждый шаг сохранять на бэкенд или минимум в `CloudStorage`.

**Не использовать горизонтальные свайпы у краёв**  
`overflow-x: scroll` без `padding: 0 20px` на iOS закрывает Mini App. Это не баг — системное поведение, не поддаётся override.

**Не просить разрешения при старте**  
`requestContact()` или `requestLocation()` при открытии — пользователь отклоняет, так как не понимает зачем. Запрашивать только в контексте конкретного шага.

**Не использовать кастомные алерты и модалки для критичных действий**  
Кастомный HTML-попап для «Вы уверены, что хотите отменить запись?» выглядит ненадёжно. Использовать `showConfirm()` — нативный диалог Telegram, которому пользователь доверяет.

**Не игнорировать тёмную тему**  
Hardcode `background: #ffffff` → в тёмной теме Telegram белый квадрат. Критично выглядит как сломанное приложение. ~90% активных пользователей Telegram имеют тёмную тему.

| Слой | Выбор | Причина |
|---|---|---|
